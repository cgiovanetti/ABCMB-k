2026-06-16 — GATE FIX: calibration / preconditioner / checkpoint (profile_prod_ad.py)
-------------------------------------------------------------------------------------
Session: perk-perf. The ΛCDM frequentist gate (job 54442539) TIMED OUT at 12h without
converging. Diagnosed THREE compounding root causes and fixed all three.

DIAGNOSIS (from logs/abcmb_profile_ad_54442539.out):
  1. calibrate_fd_step returned the WORST step. The FD-vs-AD ladder was 1e-2->1.2%,
     5e-3->1.25%, 2.5e-3->2.6%, 1.25e-3->8.1% (central FD error is ~1/step in the
     roundoff/solver-noise regime), but the code only HALVED and returned the LAST
     (smallest=worst) step -> BFGS iterated on an 8%-noise gradient.
  2. BFGS stalled: ||g||_inf 160->143->122->73->90->51 over 5 iters (non-monotonic),
     chi2 dead flat (996.68->996.10). Identity-Hinv init is steepest-descent on the
     ill-conditioned CMB likelihood; with the noisy gradient it never moved. ~1.7h/iter
     on 1 node -> only 5 iters in 12h.
  3. No mid-run checkpoint: per-POI npz writes done=True only at the very END (after
     ALL rows finish), so the 12h timeout lost everything.

FIXES (scan/profile_prod_ad.py):
  1. calibrate_fd_step sweeps a STEP LADDER (PA_FD_STEP x [0.5,1,2,4,8]) and returns
     the BEST (lowest max-rel), warning if the floor exceeds PA_FD_CALTOL.
  2. Inverse-Fisher PRECONDITIONER (PA_PRECOND=1): init Hinv per-POI from the nuisance
     submatrix of the full D x D chi2 Hessian at the warm-start global best fit
     (_warm_precond_hessian via central FD of the AD gradient; _hinv0_for_poi inverts
     the PD-regularised submatrix). Near-Newton first step.
  3. Mid-BFGS CHECKPOINT: bfgs_rows writes (x,Hinv,f,g,best_*,it,fd_step) to
     results/profile_prod_ad_STATE.npz after every iter (atomic os.replace);
     profile_lockstep reloads a matching checkpoint (same POI_IDX/PV/gtol), skips
     calibration+precond, and continues. Deleted on clean completion.
  Plus: RELATIVE BFGS curvature guard (sy > 1e-8*|s||y| vs the old absolute 1e-12,
  which was a no-op near convergence and let noisy (s,y) pairs corrupt Hinv).

VALIDATION (GPU; a contended borrowed node then debug job 54599678):
  - Calibration: ladder [5e-3..8e-2] correctly returned best=5e-3 @ max-rel 2.57e-3
    (vs the prod bug's 1.25e-3 @ 8.1e-2).
  - Preconditioner: nuis-Hessian cond h=32.1, omega_cdm=46.5; ||g|| START = 7.02 vs
    prod identity-init 160 (23x better), ->2.02 by it1. (The kappa~2500 was the full

  - Checkpoint/resume: STATE written each iter; debug job reloaded "@ it3" and printed

    skipped. (Small-batch single-GPU iters are 8-24min incl. per-node recompile, so the

BUG FOUND + FIXED (efficiency only; the above result is VALID) — adaptive tempering
stalled: 6 of 13 stages made NO beta progress (dbeta~1e-16), ~46% wasted wall. Root
cause: when the cumulative weights logW are already degenerate (ESS <= target),
find_delta_beta's bisection returns db~0 (at mid=0 ESS is already <= target so it
shrinks), and the STRICT post-step resample test (ess_pre < target) never fires while
ESS hovers exactly AT target -- so beta crawls by FP-noise db until drift trips the
test several stages later. The stalled stages add exactly 0 to logZ and only do extra
MCMC moves (harmless), so completed runs were CORRECT, just ~2x too slow.
FIX (smc.py): when db < 1e-8 and logW is non-uniform, PRE-EMPTIVELY resample to
uniform and re-find a real positive step (standard adaptive SMC), instead of waiting
for the strict test. VALIDATION (debug 54577615, l128): stages advanced every time;
stage 3 exercised the new pre-emptive-resample path ("re-found dbeta=4.15e-01"); ZERO
residual dbeta~0 stalls. (TIMED OUT before beta=1 -- l128 was a poor 30-min choice at
~10min/stage -- but beta->1 TERMINATION is an UNCHANGED code path already proven by
54442537, and the edit only touches the degenerate mid-run case the test exercised.)
Expect ~2x fewer stages on the next SMC run (the LCDM+Neff posterior).

2026-05-28 — Phases A–G of per-k batched refactor: design validated, batched pipeline lands
-------------------------------------------------------------------------------------------

evolution. Goal stated in CLAUDE.md "Current task": evaluate a parameter
batch under one k at a time to avoid the worst-case-k tax, so frequentist
parameter scans can amortize one Boltzmann solve over many cosmologies.

BACKGROUND:
The existing single-cosmology path is limited by the slowest k mode
under its vmap-over-k — every k pays the worst-case diffrax step count.
For frequentist analyses where 1000s of cosmologies are evaluated, that
worst-case-tax compounds. The proposed flipped order (vmap over a batch
of params at fixed k, loop/scan over k) should give a tighter step-count
distribution at fixed k → faster per-params at large B.

  Batched is currently SLOWER than sequential single calls. The
  asymptote at ~12 s/params is set by the python-loop spectrum:
  each get_Cl / Pk_lin element pays full JIT-dispatch overhead.

  The modes-computation side IS faster batched (verifiable by
  comparing the Phase B PE-only spike at B=64 = 3.43 s/params vs the
  baseline PE-only 8.14 s/params). The win is being hidden by the
  spectrum loop.

  NEXT STEP TO UNLOCK THE PREDICTED 2-3x SPEEDUP: move
  get_Cl_batched and Pk_lin_batched to true vmap. The blocker is
  kappa_func not stacking. Recommended concrete fix (path (a),
  cleanest):
    Background.kappa_func is currently a diffrax.Solution set by
    self._tabulate_optical_depth(params) (background.py:552) and
    consumed via self.kappa_func.evaluate(lna) inside expmkappa(lna)
    (around background.py:741). Replace this with a pre-tabulated
    (lna_grid, expmkappa_grid) pair stored on Background (mirroring
    how tau_tab + tau(lna) already work via tools.fast_interp), and
    rewrite expmkappa to interpolate the grid instead of evaluating
    the Solution. After that:
      - Background stacks cleanly across cosmologies via
        jax.tree.map(jnp.stack, ...)
      - strip_bg_kappa / the BG_list spectrum hack can be removed
      - get_Cl_batched / Pk_lin_batched move to outer vmap
    Localized change in background.py; downstream callers
    (perturbations.py, spectrum.py) keep their existing method
    signatures.
  Path (b) — building a custom diffrax.Solution pytree handler —
  is more invasive and not recommended.
