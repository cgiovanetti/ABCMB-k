ABCMB-k Change Log
==================

Each entry is written by a Claude Code session and describes what changed
and why. Entries are in reverse chronological order.

This checkout (ABCMB-k) is a per-k batching playground for the parent
PyPI-published ABCMB (TonyZhou729/ABCMB). The standalone identity is
documented in CLAUDE.md.


2026-05-29 (round 3) — Per-call GPU memory: kill the 21 GB conformal-time peak + the modes triple-buffer
---------------------------------------------------------------------------------------------------------
Session: perk-perf. Goal (user): reduce per-call GPU memory in general AND for
massive neutrinos, and improve massive-ν runtime. Two memory wins landed; the
massive-ν RUNTIME lever turned out to be a no-op and was reverted.

THE HEADLINE (A100-80GB, single GPU, kchunk=100, ELLMAX=800, lensing=False;
bench/round3_baselines.txt):
    config           baseline      after       reduction
    massless B=64    21.08 GB      9.46 GB      2.23x
    massive  B=16     9.51 GB      5.13 GB      1.85x
Runtime unchanged by both fixes.

WHAT LANDED (two independent, accuracy-neutral fixes):

1. SaveAt(ts) in _tabulate_conformal_time (background.py) -- THE BIG ONE, PORTABLE.
   The conformal-time solve used SaveAt(dense=True) then evaluated the dense
   Solution at all 10000 lna_tau_tab points via vmap. Under the B-vmap of
   background construction, sol.evaluate broadcast each query against the solver's
   padded (max_steps=4096) step array -> a (B,10000,4096) transient = 21.0 GB at
   B=64. This -- not the modes tensor -- was the BINDING peak of the whole pipeline,
   and it is Ny-INDEPENDENT (so it capped massless and massive equally). Localized
   with bench/profile_buildbgs.py (_pre_recomb_batched = 21.27 GB, get_BG only 0.4)
   and bench/runtime_peak.py (it's real runtime, not compile scratch). Fix: save
   directly at the in-domain grid suffix with SaveAt(ts=lna_tau_tab[i0:]); same
   per-step interpolant, so tau is unchanged. Gate: massless vs CLASS byte-identical
   (TT 0.1967/EE 0.2309/Pk 0.1845); snapshots 5/5 pass at rtol=1e-5 incl.
   lcdm_massive_nu (the species-independent bit-neutrality proof). PORTS to plain
   ABCMB unchanged (same code) -> ../ABCMB_memory_reduction.md.

2. transpose-kill in _compute_modes_batched (perturbations.py) -- MASSIVE-specific.
   The batched modes builder held the per-chunk list AND the concatenated
   (N_k,B,Nlna,Ny) tensor AND its transpose at once (~3x the raw modes tensor).
   Replaced with a single pre-allocated (B,Ny,Nlna,N_k) buffer that each chunk is
   transposed into and scattered into in place via a donated _write_chunk
   (donate=all-except-first). Bit-identical; validate_keystone batched-vs-single
   unchanged at the chunking-noise floor (TT |d|/peak=3.2e-7). This is the lever
   for massive (modes tensor is binding only at Ny~100): massive 9.51->5.30; then
   fix 1 took it 5.30->5.13. Massless's modes tensor was never binding, so fix 1
   alone gives its 21.08->9.46.

REVERTED (measured no-op): tabulating aH to avoid the ~5-6 rho_tot recomputes per
perturbation RHS step. XLA already CSEs the redundant aH calls within a trace, so
it gave ZERO speedup on either species (massless 1.98->2.02, massive 6.51->6.57).
(Also: interpax.interp1d(cubic) rebuilds an O(10000) spline per call = a 10%
regression; linear fast_interp is C0 and makes the rtol-1e-10 background solves hit
max_steps. None of it mattered since the recompute wasn't the bottleneck.) Massive-
nu per-solve time is the O(Ny^2) implicit Kvaerno5 solve, whose only lever is
l_max_ncdm (an accuracy trade, banned). So the massive-nu THROUGHPUT win is the
memory reduction -> fit larger B -> better amortization (massive now fits B=64
single-GPU at ~20 GB).

Also: corrects the round-2 memory model, which attributed the peak to "the
persistent saved-trajectory tensor ~3.65*N_k*Nlna*Ny*B_local". Stage isolation
shows the real binding peak was the conformal-time dense-evaluate (Ny-independent),
which round-2 mis-fit because it happened to scale ~0.33 GB/B_local too.

bench artifacts: round3_baselines.txt (all numbers), profile_buildbgs.py,
runtime_peak.py, profile_peak.py (--stop per-stage), accuracy_gate.py (vs-CLASS;
massless solid, massive setup still mismatched), mem_throughput_sweep.py (+--nlna).


2026-05-29 (round 2) — Throughput for a frequentist scan: raise B + sharding, memory model, multi-node harness
-------------------------------------------------------------------------------------------------------------
Session: perk-perf. Goal: much higher capacity for a frequentist scan over many
cosmologies — "batch B≫64 at ≤1 s/param, or ≪1 s/param." A 4-agent battle royale
(bench/round2_{solver,precision,scaleout,memory}.md) brainstormed; the user pruned
the tree with three steers, all recorded:
  - NO TCA (diffrax regime-switching is expensive — would slow us down).
  - NO accuracy-for-speed trades (stay near permille; architectural wins only).
  - NO SLURM job arrays (Perlmutter touchy; only ~2 queued jobs get priority).

THE HEADLINE (all accuracy-neutral, ZERO numerical change):
Memory — not the solver — was the throughput cap, and it dissolves under sharding.
Measured 4-GPU A100 (ELLMAX=800, lensing=False, post-compile), bench/round2_sweep.jsonl:
    B     per-param   peak/device   cosmo/s/node
    64     1.091 s      5.3 GB         0.92
    128    0.721 s     10.6 GB         1.39
    192    0.606 s     15.8 GB         1.65
    256    0.533 s     21.1 GB         1.88
    384    0.478 s     31.6 GB         2.09
    512    0.444 s     42.2 GB         2.25      (still falling, flattening;
                                                  kc=48 marginally faster, 0.434)
So just RAISING B (free, sharded) took per-param 1.09 → 0.43 (2.5×). The GPUs are
A100-SXM4-80GB, so B can go further (B_local ≲ 200 → B ≲ 800 on 4 GPUs).

MEMORY MODEL (calibrated, the key correction to the pre-session memos):
peak/device ≈ 3.65 · N_k · Nlna · Ny · 8B · B_local  (B_local = B/n_dev). It is the
PERSISTENT saved-trajectory tensor ∝ B_local, and is INDEPENDENT of k_chunk —
matches massless ΛCDM (Ny=46 → 0.33 GB/B_local) and one massive ν (Ny≈83 → 0.60
GB/B_local) exactly. The "transient ∝ k_chunk×B_local" model from the memos is
WRONG: 256@kchunk100 and 512@kchunk48 sit on the same B_local line.

k_chunk IS A THROUGHPUT KNOB, NOT A MEMORY KNOB (data, both species):
shrinking k_chunk never lowers the peak and only adds launch overhead —
  massless B=256: 0.533 s (kc=100) → 0.575 s (kc=48), peak 21.1 GB both;
  massive  B=16:  6.47 s (kc=100) → 8.22 s (kc=25),  peak 9.5 GB all.
So the proposed "adaptive k_chunk to fit big B" is refuted: to fit larger B you
SHARD (lower B_local) — k_chunk can't help. Default 100 is optimal.

MASSIVE-ν caveat (user-flagged before declaring victory): one massive ν grows
Ny 46→~83 → ~1.8× the memory per B_local (B_local ≲ 110 on 80 GB) AND is ~6×
slower per solve (B=16 single-GPU: 6.5 s/param vs ~1 s massless sharded). So
massive-ν scans need more sharding and more wall-clock. (bench/round2_massive.jsonl)

REJECTED levers (measured, not assumed): tol-loosening (3e-4/3e-6 → EE 0.39% >
permille; 1e-3/1e-5 → TT 1.03% gate fail; bench/tol_bracket_results.json);
float32/fp32-storage (no-TCA stiff term + make_output_table fp64 sums); persistent
compile cache (works for plain jit but ABCMB call_batched writes 0 entries — HyRex
custom_vjp path — so it's best-effort; one-time ~150 s/job compile is parallel
across nodes, negligible vs hours of compute).

WHAT LANDED (code, perk-perf):
1. abcmb/main.py call_batched: threaded k_chunk_size (default 100, throughput
   knob) + an Ny-aware memory WARNING (estimates peak, says "shard / lower B",
   never silently shrinks k_chunk). Behaviorally identical to the prior default
   path (verified: B=8 auto-vs-explicit-100 Δ=0; kc=100 vs 25 agree 1e-6
   peak-normalized — the documented chunking-noise floor).
2. scan/ — multi-node scan harness (the capacity answer): ONE multi-node SLURM job
   (NOT a job array), srun one independent worker per node (SLURM_PROCID), each
   shards its grid slice over the node's 4 GPUs; embarrassingly parallel, no
   jax.distributed. Resumable (skips done slices) + best-effort persistent compile
   cache. scan/scan_slice.py + scan/scan_multinode.slurm. Throughput ≈ K × per-node.

RECOMMENDATION: B≈512 per 4-GPU node (≈0.43 s/param, 42 GB/80), scale nodes with
scan/scan_multinode.slurm. Massless ≈ 2.3 cosmo/s/node → e.g. 8 nodes ≈ 18 cosmo/s.

bench artifacts: round2_plan.md (synthesis), round2_{solver,precision,scaleout,
memory}.md (battle-royale memos), round2_sweep.jsonl + round2_massive.jsonl
(measurements), mem_throughput_sweep.py + run_sweep.sh + run_massive.sh (harness),
tol_bracket.py/.json (why tol-loosening is rejected), cache_probe.py/cache_diag.py
(cache finding), validate_autokchunk.py (parity).


2026-05-29 — Performance pass on call_batched: 12.4 → ~1 s/param (target met)
---------------------------------------------------------------------------
Session: perk-perf branch (cut from perk-refactor). Goal: make the already-
landed batched pipeline FAST — no new features, just performance. Target
~1 s per parameter combination.

THE KEY REFRAMING (from measurement, not assumption):
The Phase A baseline said the perturbation solve dominates (8.1 of 9.9 s
single-cosmology). True single-cosmology, but FALSE in the batched path. A
real per-stage block_until_ready profile of Model.call_batched
(bench/profile_stages.py) showed:
  - perturb amortizes well: 12.2 (B=1) → 2.6 (B=8) → 2.0 (B=16) s/param.
  - the bottleneck was TWO FLAT eager-dispatch costs that did NOT amortize:
      spec_Cl  ~3.2 s/param  (the python-loop spectrum)
      setup    ~6.9 s/param  (get_BG run EAGERLY, not jitted)
  - end-to-end batched was ~12.4 s/param at B=16 — SLOWER than 9.5 single.
The whole win came from killing those two flat costs, not from tuning the
solver. (A k_chunk_size sweep confirmed the default 100 is already optimal.)

WHAT LANDED (commits on perk-perf):

1. Keystone — tabulate expmkappa, vmap the spectrum (63696b9):
   Background.kappa_func was a diffrax.Solution — the one non-array leaf that
   blocked stacking Backgrounds across cosmologies, forcing get_Cl_batched /
   Pk_lin_batched into python loops (each get_Cl ran fully EAGER → the flat
   3.2 s/param). Replaced it with expmkappa_tab, a plain array tabulated on
   the shared lna_tau_tab axis (mirroring tau/tau_tab), and rewrote expmkappa
   to interpolate it. Background is now a pure-array PyTree that stacks
   cleanly, so get_Cl_batched / Pk_lin_batched became a single
   @eqx.filter_jit jax.vmap over B, and strip_bg_kappa was deleted.
     spec_Cl: 3.18 → 0.01 s/param (~300x).

2. Cubic interp for expmkappa (d194627):
   Linear fast_interp made expmkappa C0, so grad(BG.visibility) (the ClTE
   source term) was piecewise-constant and amplified diffrax step-noise into
   ~1e-1 ClTE error. Switched to interpax.interp1d(method="cubic") (C1). Only
   TE was affected (TT/EE/Pk were already at the ~3e-4 diffrax-noise floor);
   the 10000-pt grid makes the cubic essentially exact vs the old dense
   Solution.evaluate.

3. Batch + shard the setup (ed1224c):
   Replaced the per-cosmology python _build_one_bg loop (get_BG run EAGERLY →
   the flat ~7 s/param) with _build_bgs_batched: vmapped BackgroundPreRecomb
   (GPU), vmapped RecModel/HyRex (CPU), vmapped get_BG (GPU) — one jit + one
   device transfer each way instead of O(B). Possible only because of (1).
   Added call_batched(shard=): jax.sharding Mesh + NamedSharding(P('batch'))
   applied BEFORE the setup so every GPU stage (pre-recomb, get_BG, perturb,
   spectrum) auto-partitions over the mesh and each device builds/solves only
   B/n_dev cosmologies (GSPMD, no collectives; embarrassingly parallel over
   B). B is padded to a multiple of the device count and the padding sliced
   off. (Sharding AFTER the setup instead built all B on device 0 → OOM at
   B=64; fixed by sharding first.)

PERF (ELLMAX=800, A100, post-compile, per param):
              before    single-GPU after    4-GPU sharded
   B=8         12.6            6.54              —
   B=16        12.4            4.00            3.28
   B=32         —              1.81            1.81
   B=64         —               —             1.13
  single-GPU B=16 stage split after: setup 2.01 + perturb 1.97 + spec 0.01;
  setup amortizes 30.8 → 2.0 as B grows, so perturb is now the single-GPU
  floor. 4-GPU B=64 = 1.13 s/param meets the ~1 s/param goal and is still
  falling with B (sharding quarters per-device memory, so larger B fits).

CORRECTNESS (gates both PASS — unchanged by the refactor):
  - pytests/accuracy_test.py (1%-vs-CLASS, single path): TT 0.197%, EE 0.231%,
    Pk 0.185% — all < 1%.
  - batched-vs-single Cl/Pk peak-normalized |Δ|/peak ~1e-6 (better than the
    pre-change ~1e-5 python-loop). The pointwise-relative metric is meaningless
    across ClTE zero-crossings (|spectrum|→0); use peak-normalized error
    (bench/validate_keystone.py).
  - snapshot fixtures regenerated: the keystone shifts ABCMB output at the
    ~1e-8 snapshot tolerance (interp vs dense Solution.evaluate) — deliberate.

NEW/CHANGED bench artifacts: profile_stages.py (per-stage profiler — the
experiment that reframed the problem), perf_multigpu.py (multi-GPU sharded
benchmark + pad-check), sweep_kchunk.py (k_chunk=100 confirmed optimal),
probe_setup.py (eager get_BG = 5.5x slower than jitted), validate_keystone.py
(peak-normalized parity). Analysis memos: bench/notes_{spectrum,solve,strategy}.md
and bench/ideas_{singlegpu,multigpu,redteam}.md (two subagent brainstorm rounds).


2026-05-28 — Phases A–G of per-k batched refactor: design validated, batched pipeline lands
-------------------------------------------------------------------------------------------
Session: perk-refactor — first pass on the flipped (params-first) batched
evolution. Goal stated in CLAUDE.md "Current task": evaluate a parameter
batch under one k at a time to avoid the worst-case-k tax, so frequentist
parameter scans can amortize one Boltzmann solve over many cosmologies.

BACKGROUND:
The existing single-cosmology path is limited by the slowest k mode
under its vmap-over-k — every k pays the worst-case diffrax step count.
For frequentist analyses where 1000s of cosmologies are evaluated, that
worst-case-tax compounds. The proposed flipped order (vmap over a batch
of params at fixed k, loop/scan over k) should give a tighter step-count
distribution at fixed k → faster per-params at large B.

WHAT LANDED ON THE perk-refactor BRANCH:

Phase A — baseline profile (bench/baseline.py, .npz, _summary.txt):
  Single-GPU, fiducial-perturbed ΛCDM box (Planck ±2-3σ on
  h, omega_cdm, omega_b, A_s, n_s).
    - End-to-end model() per-params: 9.89 s
    - PE.full_evolution per-params:  8.14 s
    - Per-k step counts (N_k=571): min=41 median=380 max=1579
    - max/median = 4.16 — that's the worst-case-tax to beat.

Phase B — flipped-order spike (bench/flipped_spike.py, multigpu, _summary.txt):
  Three iterations:
    1. Original lax.scan-over-k around vmap-over-batch around diffrax:
       13-min compile + 13-min run per B value → abandoned.
    2. Python-loop-over-k, vmap-over-batch: structurally guaranteed slow
       because each k call serializes → abandoned.
    3. Double-vmap (vmap-over-k × vmap-over-batch): WORKS.

  Single-GPU at B=64:
    - per-params 3.43 s = 2.37x speedup vs baseline 8.14
    - memory pressured (28-31 GB on A100 40 GB; XLA rematerialization)

  4-GPU sharded over batch axis (bench/flipped_spike_multigpu.py):
    - per-params at B=64: 0.93 s = 8.80x speedup vs baseline

  Step-count hypothesis CONFIRMED:
    - baseline max/median over k = 4.16
    - flipped worst-k max/median over B = 1.12

  Gate evaluator (bench/decision.py) verdict: PROCEED. Confirming that
  the design pays out at meaningful B, and multi-GPU sharding restores
  linear scaling once memory bites.

  CAVEAT (noticed only post-hoc, during the chunking-debug closeout):
  The Phase B spike measured wall-clock only, not per-mode trajectory
  parity. The 8.80x at B=64 multi-GPU is a correct wall-clock number
  but the trajectories the spike produced were subject to the same
  diffrax step-controller noise documented later in
  bench/chunking_debug_report.md — i.e., they don't bit-match single-
  call full_evolution. The right contract is downstream Cl/Pk parity,
  which smoke_batched_pipeline.log confirms at ~1e-5 rel. Anyone re-
  running the spike and expecting per-mode bit-parity will be confused.

Phase C — snapshot fixtures + parity helpers (pytests/):
  Four scenarios: lcdm_tau, lcdm_z, lcdm_massive_nu, bbn_table.
  (bbn_linx skipped — slow CPU pipeline, can add later.)
  Helpers: pytests/snapshots.py (assert_matches_snapshot),
  pytests/parity.py (assert_batch_matches_loop).
  Snapshot tolerance set to rtol=1e-8, atol=1e-18 — tighter rtol turned
  out to be optimistic given XLA's run-to-run scheduler variability on
  imports of new methods. The previous 1e-10 was below what XLA
  actually delivers across code edits.

Phase D — batched perturbation evolver:
  abcmb/perturbations.py adds:
    - _evolve_chunk (@eqx.filter_jit): vmap over k_chunk × vmap over B
      around evolution_one_k. Reused per-shape JIT cache entry.
    - _compute_modes_batched: chunked python loop over k of _evolve_chunk
      with k_chunk_size=100 default. Memory ≈ 5.5 GB at B=64 on A100.
    - make_output_table_batched (@eqx.filter_jit): vmap'd outer
      composition of the existing make_output_table.
    - full_evolution_batched: orchestrator returning batched
      PerturbationTable.
    - strip_bg_kappa (module-level): replaces Background.kappa_func with
      None so jax.tree.map(jnp.stack, ...) on a batch of Backgrounds
      succeeds. Perturbations don't touch kappa_func.

Phase E — batched spectrum (abcmb/spectrum.py):
  SpectrumSolver gains get_Cl_batched and Pk_lin_batched. CURRENTLY
  IMPLEMENTED AS PYTHON LOOPS over B because get_Cl reads
  BG.kappa_func (via visibility / expmkappa), and kappa_func doesn't
  stack across cosmologies cleanly. This is the major remaining perf
  bottleneck (see below).

Phase G — Model.call_batched + BatchedOutput (abcmb/main.py):
  Model.call_batched(params_list) is the user-facing entrypoint.
  Internally:
    1. Build B Background objects sequentially via _build_one_bg
       (HyRex on CPU, no batching there yet — design memo justifies
       deferring).
    2. Stack params + strip-and-stack BGs for the modes computation.
    3. full_evolution_batched on the stripped BG batch → batched PT.
    4. Spectrum on a python list of un-stripped BGs (kappa_func intact).
  BatchedOutput stores ClTT/ClTE/ClEE/Pk (shape (B, l)/(B, k)) plus l, k,
  and the batched params dict. Background and PerturbationTable are
  intentionally NOT carried — kappa_func doesn't stack.

PARITY:
  End-to-end smoke (bench/smoke_batched_pipeline.py) at B=2 matches
  single model() FIDUCIAL on Cls/Pk to:
    ClTT max_rel = 1.42e-07
    ClTE max_rel = 2.97e-05
    ClEE max_rel = 1.59e-08
    Pk   max_rel = 2.59e-05
  All under the 1% accuracy_test.py gate. accuracy_test.py against CLASS
  has not been run on the batched path yet — listed as next step.

THE FALSE "CHUNKING BUG":
  Mid-session, _compute_modes_batched showed wildly different numerical
  output between chunked and unchunked paths (max_rel up to 1e+5).
  Investigation took several hours. A subagent (12 diagnostic smokes in
  bench/smoke_chunk*.py + smoke_full_vs_pure.py + smoke_convergence.py)
  ultimately diagnosed:
    THIS IS NOT A BUG. It is diffrax's PID adaptive step controller
    behaving batch-composition-dependently under vmap. Different
    k-mode batches → different worst-case step counts → different
    actual stepping → trajectories disagree at the per-mode level but
    all stay within the configured rtol_large_k_PE = 1e-4 envelope.

  Hard evidence (bench/chunking_debug_report.md):
    The "max_abs 8.6e-1 between chunk and full-vmap" was on
    quantities of norm ~1e4 — i.e., ~1e-5 relative, fine for spectrum
    integration. smoke_convergence at rtol_large=1e-4 shows both
    full-vmap and chunked agree with pure-single-k integration to
    ~1e-6 relative. Tighter rtol brackets confirm convergence.

  Implications:
    - Chunking works; the spectrum/PT downstream agreement is the
      contract that matters (already at 1e-5 in
      smoke_batched_pipeline.log).
    - We rolled back the python-loop workaround in
      make_output_table_batched after this diagnosis.

PERFORMANCE (bench/perf_batched.py, ELLMAX=800, single A100):
    single (post-compile):         9.51 s/params
    batched B=1:                  22.64 s   (0.42x vs single×B)
    batched B=4:                  14.46 s/params  (0.66x)
    batched B=8:                  12.95 s/params  (0.73x)
    batched B=16:                 12.08 s/params  (0.79x)

  Batched is currently SLOWER than sequential single calls. The
  asymptote at ~12 s/params is set by the python-loop spectrum:
  each get_Cl / Pk_lin element pays full JIT-dispatch overhead.

  The modes-computation side IS faster batched (verifiable by
  comparing the Phase B PE-only spike at B=64 = 3.43 s/params vs the
  baseline PE-only 8.14 s/params). The win is being hidden by the
  spectrum loop.

  NEXT STEP TO UNLOCK THE PREDICTED 2-3x SPEEDUP: move
  get_Cl_batched and Pk_lin_batched to true vmap. The blocker is
  kappa_func not stacking. Recommended concrete fix (path (a),
  cleanest):
    Background.kappa_func is currently a diffrax.Solution set by
    self._tabulate_optical_depth(params) (background.py:552) and
    consumed via self.kappa_func.evaluate(lna) inside expmkappa(lna)
    (around background.py:741). Replace this with a pre-tabulated
    (lna_grid, expmkappa_grid) pair stored on Background (mirroring
    how tau_tab + tau(lna) already work via tools.fast_interp), and
    rewrite expmkappa to interpolate the grid instead of evaluating
    the Solution. After that:
      - Background stacks cleanly across cosmologies via
        jax.tree.map(jnp.stack, ...)
      - strip_bg_kappa / the BG_list spectrum hack can be removed
      - get_Cl_batched / Pk_lin_batched move to outer vmap
    Localized change in background.py; downstream callers
    (perturbations.py, spectrum.py) keep their existing method
    signatures.
  Path (b) — building a custom diffrax.Solution pytree handler —
  is more invasive and not recommended.
